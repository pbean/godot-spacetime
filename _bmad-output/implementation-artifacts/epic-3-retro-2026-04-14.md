## Successes and Strengths

### 1. Overlap-First Subscription Replacement (Story 3.4)

The overlap-first design in Story 3.4 is the standout architectural achievement of the epic. The old subscription remains authoritative in the local cache until the new subscription is confirmed applied by the server. Gameplay code never observes an empty or partial cache during a scene transition. If the new subscription fails before being applied, the old one is untouched — the failure path is as clean as the success path.

**Evidence:** Story 3.4 AC 2 and AC 5. The `_pendingReplacements` map (newHandleId → oldHandleId) defers old-handle cleanup to `OnSubscriptionApplied`, ensuring atomic transition. Best-effort TryUnsubscribe prevents SDK close failures from blocking local handle state transitions.

### 2. Isolation Zone Discipline — Zero Boundary Violations

All five stories, zero `SpacetimeDB.*` references outside `Internal/Platform/DotNet/`. The Expression-tree adapter pattern established in Epic 2 (auth callbacks) was reused for subscription adapters (Story 3.1) and row callback adapters (Story 3.3) without modification. The architecture is being executed consistently.

**Evidence:** Story 3.1, 3.2, 3.3, 3.4, 3.5 completion notes — all verify "Does NOT contain `SpacetimeDB.` string" as explicit test assertions.

### 3. Stub-First Infrastructure Pays Dividends (Stories 3.4 and 3.5)

The `ISubscriptionEventSink.OnSubscriptionError` stub from Story 3.1 was intentionally left as a no-op with a "Story 3.5 will surface this error" comment. Story 3.5 implemented the full failure surface by adding three targeted changes: `SubscriptionFailedEvent` payload, `OnSubscriptionFailed` service event, and `SubscriptionFailed` Godot signal. No structural churn, no rework of the underlying infrastructure.

Story 3.5 was the most efficient delivery in the epic because Epics 1–3 had correctly built the plumbing without prematurely implementing the surface.

### 4. Test Suite Growth and Baseline Quality

The test suite grew from 665 to 1148 — a 73% increase across the epic. Each story added story-specific contract tests plus regression guards covering all prior stories. The `pytest -q` full-suite gate caught no regressions across the epic.

### 5. Field-Per-Feature Pattern Scales Well

`_cacheViewAdapter`, `_rowCallbackAdapter`, `_pendingReplacements` — each story added exactly its needed component as a `private readonly` field in `SpacetimeConnectionService`. No story required reworking another story's fields. The pattern absorbs new complexity without structural churn.

## Challenges and Growth Areas

### 1. Non-Happy-Path Exit Coverage — The Confirmed Systemic Gap

Four HIGH findings across five stories, every one in the same category: state cleanup on secondary lifecycle paths.

| Story | HIGH Finding | Secondary Path Missed |
|-------|-------------|----------------------|
| 3.2 | `CacheViewAdapter` not cleared on failed/lost connections | Failed connect, degraded→lost connection |
| 3.3 | Row callback re-registration on every `OnConnected` | Connected → Degraded → Connected recovery |
| 3.3 | Row callbacks wired only on properties, not fields | SpacetimeDB 2.1.0 uses public fields for table handles |
| 3.4 | SDK `TryUnsubscribe` reflection failures not caught | Reflection invocation errors abort before handle cleanup |

The pattern: implementations correctly handle the primary path (explicit disconnect, normal subscribe) but miss secondary paths (failed connects, degraded reconnect recovery, reflection invocation failures). This is the same category as both HIGHs in Epic 2 (ProjectSettings.Save() not called; restored credentials not cleared after Open()).

**Root cause:** Secondary lifecycle paths are not systematically addressed during story development. Dev notes document the happy path in detail but do not enumerate "what must be cleaned up in ALL exit paths."

**Impact:** None of these HIGHs would have been caught by `pytest -q`. Static contract tests verify code exists — they cannot verify that cleanup code runs in all expected circumstances. The senior review step caught all four, but this is a systemic gap, not a one-off miss.

### 2. Generated Bindings Inspection Required — Not Just Documentation

Story 3.3's most significant finding: `SpacetimeSdkRowCallbackAdapter` iterated public *properties* on generated `RemoteTables`, but SpacetimeDB SDK 2.1.0 exposes table handles as public *fields*.

The row callback adapter as initially implemented would never have fired `RowChanged` for any real generated client — because the actual generated bindings use fields, not properties. This bug was invisible until the senior reviewer checked `demo/generated/smoke_test/Tables/SmokeTest.g.cs` directly.

**Root cause:** API inspection stopped at documentation and interface signatures. The generated bindings in `demo/generated/` are the authoritative source of truth for SDK shape — not documentation.

**Impact:** Had this shipped without the senior review catch, every game using this SDK would have had `RowChanged` events that silently never fired. A fundamental contract of the subscription lifecycle would have been broken.

### 3. Property Mutability Gap (Story 3.1)

`SubscriptionHandle.HandleId` and `SubscriptionAppliedEvent.Handle`/`AppliedAt` were implemented as `get; private set;` instead of genuinely getter-only properties. The story's acceptance criteria established a "stable handle" and "immutable event" guarantee; the implementation allowed internal mutation. Three MEDIUM findings resolved by changing to getter-only.

**Root cause:** Developers defaulted to `get; private set;` (common C# pattern) rather than matching the immutability intent of the story spec.

### 4. Public Documentation Lag (Stories 3.2 and 3.5)

Story 3.2's MEDIUM finding: `SpacetimeClient` XML docs and `runtime-boundaries.md` still described cache reads "via generated bindings" instead of the new `GetRows()` boundary after the story shipped.

Story 3.5's MEDIUM finding: `runtime-boundaries.md` and public XML docs omitted the `SubscriptionFailed` event entirely after the signal was added.

**Root cause:** Documentation updates are treated as a task item to check off but not verified for completeness against the full public surface.

### 5. Five Epic 2 Action Items Rolled Over — Again

## Key Insights and Learnings

1. **The non-happy-path exit pattern is the confirmed systemic quality gap.** Four HIGHs in this epic, two HIGHs in Epic 2 — all in the same category. Secondary lifecycle paths (failed connect, degraded reconnect, reflection errors) must be explicitly enumerated and covered in every stateful story.

2. **Static contract tests cannot catch runtime lifecycle correctness.** This is now a two-epic confirmed pattern. 1148 tests cannot catch whether `SetDb(null)` runs on a failed connect or whether callbacks are re-added on reconnect. The test strategy must include at least one dynamic lifecycle test for any story with stateful write-through behavior.

3. **Inspect generated bindings, not just documentation.** API documentation is insufficient — the actual generated client shape (`demo/generated/smoke_test/`) must be inspected before writing any adapter. Properties in docs ≠ fields in the generated client. This is mandatory before Story 4.1.

4. **Overlap-first is the correct model for stateful replacements.** Story 3.4's approach — authorize the new path before closing the old one — eliminated cache-gap bugs without burdening gameplay code. This pattern should inform any future stateful replacement or transition operation.

5. **Stub-first architecture continues to scale.** Story 3.5 was nearly free because the plumbing was correctly laid in Story 3.1. Deferring the surface (OnSubscriptionError no-op stub) while installing the infrastructure (ISubscriptionEventSink contract, error method signature) is validated as the right pattern.

6. **Rollover items need explicit named owners, not just framing.** Items framed as "can happen in parallel" with no named owner have rolled over for three consecutive epics. Assigning a specific owner at retrospective time is mandatory — not optional — for items we want to see resolved.

## Significant Discovery Check

**No significant architectural changes required for Epic 4.** The Epic 4 plan as written is sound. The subscription infrastructure, signal dispatch pattern, and isolation zone architecture performed correctly through all five Epic 3 stories.

**Two discoveries that must inform Epic 4 implementation:**

1. **SpacetimeDB SDK 2.1.0 uses public fields (not just properties) for generated binding exposure.** Story 3.3 nearly shipped a row callback adapter that would never have fired for any real generated client. The Epic 4 reducer adapter must inspect `demo/generated/` for actual binding shape — fields AND properties — before implementing the invocation path.

2. **SDK `Subscribe()` return type may be null (void-equivalent).** Story 3.4 designed graceful degradation (`TryUnsubscribe` returns false if SDK object is null). Epic 4 reducer calls may exhibit similar nullable return behavior — design graceful degradation into the adapter from the start.

**Epic update required: NO.** Epic 4 plan is sound given the preparation tasks above are completed first.
