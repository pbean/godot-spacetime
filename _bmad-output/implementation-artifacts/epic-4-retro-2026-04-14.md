## Successes and Strengths

### 1. Zero HIGH Findings — The Exit-Path Checklist Worked

Epic 3 closed with four HIGH findings across five stories, every one in the same category: state cleanup on secondary lifecycle paths. The retrospective named it the "confirmed systemic quality gap" and created action item P1: every stateful story must enumerate all disconnect paths in dev notes before marking tasks complete.

Epic 4 applied P1 to all four stories and produced zero HIGH findings across four reviews. This is the most direct and measurable process improvement in the project's history.

**Evidence:** Story 4.1 dev notes Task 3.6 (`Disconnect path checklist — required per Epic 3 Retro action P1`) with explicit enumeration of four exit paths. Stories 4.2, 4.3, and 4.4 each carried the same checklist in their task lists. No HIGH findings appeared in any review.

### 2. Story 4.3's Teardown Guard — A Real Defect Caught Before Merge

Story 4.3's senior review found a genuine runtime correctness defect: `SpacetimeConnectionService` could recursively re-enter disconnect cleanup if `_adapter.Close()` triggered `OnDisconnected` during teardown, risking duplicate `ConnectionClosed` delivery to gameplay code.

The fix — `_isTearingDownConnection` flag and `RunConnectionTeardown()` wrapper — is not a documentation gap or style issue. It is a real race condition that would have produced intermittent duplicate connection-closed signals under specific close-sequence timing.

**Evidence:** Story 4.3 senior review tests: `test_connection_service_declares_teardown_guard_field`, `test_connection_service_has_teardown_guard_helper`, `test_connection_service_on_disconnected_ignores_teardown_and_late_callbacks`. The guard is now locked by four specific tests in the regression suite.

### 3. Story 4.2 Payload Expansion — InvocationId and RecoveryGuidance

The Story 4.2 original spec defined `ReducerCallResult` and `ReducerCallError` with `ReducerName`, timestamps, and `FailureCategory`. The senior review surfaced two missing contracts: gameplay code could not distinguish repeated in-flight invocations of the same reducer (`InvocationId`), and failure payloads did not expose an explicit user-safe recovery path (`RecoveryGuidance`).

Both were added during the review. The expanded payload surface makes the reducer result types genuinely usable for real gameplay: `InvocationId` enables correlation of concurrent reducer calls; `RecoveryGuidance` provides human-readable recovery text without requiring gameplay code to map from `FailureCategory` manually.

**Evidence:** Story 4.2 test-summary senior review section — 16 tests added covering `InvocationId`, `CompletedAt` (on result), `CalledAt` (on error), `RecoveryGuidance`, and pending-invocation tracking in the adapter.

### 4. Generated Bindings Inspection Caught Field-vs-Property Issue Proactively (P3)

Epic 3's most damaging finding was Story 3.3's field-vs-property bug: the row callback adapter iterated public *properties* on `RemoteTables`, but SpacetimeDB 2.1.0 exposes table handles as public *fields*. Had this shipped, `RowChanged` events would have silently never fired for any real generated client.

Epic 3 Retro created action P3: inspect `demo/generated/` bindings before any SDK adapter story. Story 4.1 dev notes explicitly document the P3 inspection: `RemoteReducers Reducers` is a public readonly **field** on `DbConnection` (confirmed from `demo/generated/smoke_test/SpacetimeDBClient.g.cs:586`). The Story 4.1 adapter used `GetField("Reducers")` correctly rather than `GetProperty`.

**Evidence:** Story 4.1 dev notes section "Critical: Generated Binding Shape — Inspect Before Implementing (Retro Action P3)" with direct reference to the generated file line.

### 5. Complete 8-Signal Catalog Documented

Story 4.3 added `ConnectionClosed` + `ConnectionClosedEvent`, completing the connection lifecycle bridge symmetric with `ConnectionOpened`. The story also added a "Scene-Safe Signal Bridge" section to `runtime-boundaries.md` with the full signal catalog:

| Signal | Payload | Category |
|--------|---------|----------|
| `ConnectionStateChanged` | `ConnectionStatus` | Connection lifecycle |
| `ConnectionOpened` | `ConnectionOpenedEvent` | Connection lifecycle |

This document is now the definitive runtime reference for Epic 5 documentation and sample work.

### 6. Programming Fault vs Recoverable Failure Clarity (Story 4.4)

Story 4.4 fixed the one silent swallow in the codebase (`Console.Error.WriteLine + return` in the adapter's null-connection guard) and corrected `runtime-boundaries.md` to accurately describe the programming fault model. The resulting documentation section — "Reducer Error Model: Programming Faults vs Recoverable Failures" — provides an exact branching table covering all six conditions:

- Server rejects → `ReducerCallFailed` → branch on `FailureCategory`
- Server out of energy → `ReducerCallFailed` → back off / retry
- Status unclear → `ReducerCallFailed` → handle defensively
- Called while not Connected → `ConnectionStateChanged(Disconnected)` + `GD.PushError` → fix the code
- Called with null args → `ConnectionStateChanged(Disconnected)` + `GD.PushError` → fix the code
- Called with non-IReducerArgs → `ConnectionStateChanged(Disconnected)` + `GD.PushError` → fix the code

This table feeds directly into Epic 5's troubleshooting documentation (Story 5.5).

## Challenges and Growth Areas

### 1. Story 4.1's CRITICAL: The SDK Generic Constraint

Story 4.1 was marked review-ready while the solution did not compile. `SpacetimeSdkReducerAdapter.Invoke()` called `IDbConnection.InternalCallReducer()` through an `IReducerArgs` interface value — but the SDK surface is generic (`InternalCallReducer<T>(T)` with a concrete-type constraint). The initial generated-bindings inspection identified the correct invocation pattern (`conn.InternalCallReducer(new Reducer.Ping())`) but the adapter implementation cast to the interface instead of using the concrete type.

**Root cause:** The bindings inspection noted that the generated client uses concrete types and the interface is `IReducerArgs`, but the adapter implementation still routed through the interface. The generic constraint enforcement at compile time was not checked against the local build before marking review-ready.

**Impact:** The review caught a non-compiling story. No production impact, but a review cycle was consumed for a build break that should have been caught by the developer.

**Lesson:** Bindings inspection must include a compile verification step — checking the generated pattern is not enough if the adapter's own build is not validated before sending to review. The `dotnet build` step in the pre-review checklist should be mandatory, not assumed.

### 2. Medium Finding Persistence — Docs and Tests Lagging Implementation

Four stories produced 11 MEDIUM findings. Across all four, the most common categories were:
- Docs describing behavior less accurately than the implementation (4.1, 4.3, 4.4)
- XML docs on public methods not matching the reviewed contract (4.2, 4.4)
- Tests using broad file-level string checks instead of method/section-scoped assertions (4.4)

The pattern is that implementation, docs, and test evidence are not being synchronized before review. The developer marks tasks as complete when the code is done, but the docs and test precision tend to follow at lower fidelity.

**Root cause:** No explicit pre-review checklist item for "verify docs and test assertions scope match the specific method/section being claimed." The exit-path checklist (P1) solved the HIGH finding pattern; a similar docs-precision checklist is needed.

### 3. Rollover Items — Four Epics of D1, D2/PA1, D4/PA2 Without Resolution

| Item | Epics Outstanding |

These three items have now rolled over through Epics 2, 3, and 4 without resolution. D3 (dynamic lifecycle tests for pre-Epic-4 adapters) was added in Epic 3 and has already rolled into Epic 4.

**Root cause:** YOLO-mode retrospectives assign named owners but cannot execute the follow-up. The named-owner mechanism from Epic 3 retro worked for process items (P1, P2, P3) because those were embedded in story task lists. But debt items (D1, D2, D4) have no story that "owns" them — they exist only in the retro document. Without a story that explicitly includes them as tasks, they will not get done.

**Impact:** D1 (AuthRequired dead code) is specifically problematic. It creates a confusing enum case that signals to both developers and future contributors that there is a valid auth-required path — but there isn't. As the auth and failure models grow (Epic 4 added `ReducerFailureCategory`, Story 4.4 added the programming-fault model), the dead code compounds the confusion.

D4/PA2 (editor panel verification) now directly blocks Epic 5 sample project work. Story 5.1 requires a functional addon in Godot 4.6.2 — unverified editor panels cannot remain undiscovered when the first external developer follows the sample setup path.

### 4. Story 4.3 Senior Review Found a Real Runtime Defect (Captured as Success, but Worth Understanding Why)

While the teardown guard (captured under Successes) represents a genuine quality win, the fact that it required senior review to find it warrants examination. The teardown guard pattern (`_isTearingDownConnection`, `RunConnectionTeardown()`) is not exotic — it is the same class of "what happens if the SDK fires a callback during our own cleanup" question that P1 is designed to surface.

**Root cause:** P1's disconnect-path checklist asks developers to enumerate "all exit paths" — but it frames exit paths as external triggers (explicit disconnect, connection lost, failed connect, rapid reconnect). It does not explicitly prompt: "what if the SDK fires callbacks *while* we are in teardown?" This is a subtle omission; the checklist covers entry/exit states but not re-entrant teardown.

**Lesson:** The P1 checklist needs a fifth bullet: "Re-entrant SDK callbacks during teardown: does `_adapter.Close()` trigger any SDK callbacks that could call back into this service?"

## Key Insights and Learnings

1. **The exit-path checklist (P1) is validated as the most impactful quality improvement in the project.** Four epics in, it produced the first zero-HIGH-finding epic. The mechanism is direct: enumerate the exit paths before marking tasks complete, and the state-cleanup bugs surface before review rather than in review.

2. **The generated-bindings inspection (P3) is a mandatory first step — and it must include a compile verification.** Story 4.1's CRITICAL was not a missing inspection; the inspection was done correctly. The failure was that the adapter's own build was not verified against the pattern. Add `dotnet build` as a required step in the story pre-review checklist.

3. **The senior review step catches real defects that the developer process misses.** Story 4.3's teardown guard and Story 4.2's `InvocationId`/`RecoveryGuidance` additions are both substantive improvements, not pedantic corrections. The review is not just quality-checking — it is expanding the contract in ways the original story didn't fully specify.

4. **D1 (AuthRequired dead code) must be scheduled in a story, not a retro action item.** Four retro documents have listed it. Four times it has not been done. The only mechanism that works in this project is a story task list item. Either add it as a sub-task in the first Epic 5 story, or create a dedicated preparation task that must be done before Story 5.1 is written.

5. **D4/PA2 (editor panel verification) is now blocking.** Epic 5 Story 5.1 requires a functional sample in Godot 4.6.2. The editor panels have not been verified since Stories 1.7–1.9 implemented them. This is not a "nice-to-have" for Epic 5 — it is a prerequisite for Story 5.1 to be implementable.

6. **The docs-precision gap is the remaining quality vector to address.** The exit-path gap is closed. The bindings-inspection gap is closed. The remaining systematic MEDIUM finding source is docs and test assertions lagging implementation precision. A docs-verification step in the pre-review checklist would close this gap the same way P1 closed the exit-path gap.
