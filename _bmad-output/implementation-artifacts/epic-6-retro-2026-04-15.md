## Key Insights and Lessons Learned

### Insight 1: Pipeline Design with Typed Contracts Is a Force Multiplier

The `report["status"] == "pass"` publication gate contract between Story 6.3 and Story 6.4 is the right pattern for multi-stage pipelines. It's machine-readable, explicit, versioned, and produces a paper trail. Future release pipelines should follow this contract-first design: define the interface between stages before implementing either stage.

### Insight 2: Three Failed Retro Cycles Means the Work Needs a Story

D3 (dynamic lifecycle tests) has failed to land across Epic 4, Epic 5, and Epic 6. The data is clear: retro action items and prep tasks without story anchors do not get done. The next iteration of this work needs a properly scoped story with acceptance criteria, not another carry-forward debt label.

### Insight 3: Security Posture Improves When Review Has Specific Targets

Stories 6.3 and 6.4 caught path-traversal and JSON-parsing risks because the review was looking for them. The codebase is security-adjacent (release pipeline, file system operations, subprocess calls) and the reviews reflected that. This kind of domain-aware review quality should be maintained as the project evolves.

### Insight 4: Epic 6 Completes the Build Phase

Epic 6 delivered the release pipeline. Combined with Epics 1–5, the project now has: a runtime SDK, authentication, subscriptions, reducers, documentation, sample project, and a repeatable release process. This is a complete initial release. The next phase of work — if planned — is likely adoption-driven: community feedback, compatibility expansion, or feature additions.

### Insight 5: P1 Requires Process Change, Not Just Reminders

P1 (pre-implementation test design) has improved review quality but hasn't changed when tests are written. The test suites are still written alongside implementation and expanded in review. Changing this requires a structural change — a required "test skeleton" task as step 0 in every story, before any implementation tasks.
