## Challenges and Growth Areas

### 1. Godot Editor Plugin Loading Is Underspecified (Story 1.1 — 3 spec change cycles)

The editor plugin worked as a build artifact before it could actually be enabled in the Godot editor. Two separate issues were discovered only after running the actual Godot 4.6.2 editor:

1. `plugin.cfg` paths must be addon-relative, not project-root-relative
2. The C# file name must match the plugin class name for Godot's C# editor-plugin resolution

**Root cause:** Godot's C# editor-plugin loading behavior is underdocumented. Build success does not imply editor enablement.

**Impact:** Story 1.1 required 3 spec change cycles. Subsequent editor panel stories (1.7, 1.8, 1.9) followed the same code patterns but were not independently editor-verified.

**Action item:** Add explicit editor-verification task to all future editor panel stories.

### 2. Pre-Implementation Backlog Restructure (Sprint Change 2026-04-11)

Before any code was written, the epics document had to be restructured. Runtime-neutral foundation stories (what became Stories 1.3–1.5) were originally placed in Epic 6. The implementation readiness review caught the ordering problem, but it required a dedicated sprint change spec before Epic 1 could start.

**Root cause:** Initial epic creation did not verify that architectural dependencies were sequenced correctly across epic boundaries.

**Impact:** Implementation start was delayed by the restructure. Risk: if the restructure had been missed, Epic 2–5 would have depended on foundation work that hadn't been done yet.

**Action item:** Pre-epic backlog review should include an explicit dependency order check across all epics.

### 3. Third-Party API Surface Assumed Rather Than Inspected (Story 1.4)

Architecture documentation referenced `DbConnection` as if it were a concrete class. `SpacetimeDB.ClientSDK 2.1.0` exposes `IDbConnection` (an interface). Story 1.4 caught this because the story was still structural, but if a later story had assumed `new DbConnection()`, it would have been a compilation failure mid-implementation.

**Root cause:** Architecture written before the dependency was actually inspected at the package level.

**Action item:** Any story with acceptance criteria that depend on a previously-uninspected SDK method must reference the exact confirmed API signature in its spec.

---

## Key Insights and Learnings

1. **Verify third-party API surfaces before writing acceptance criteria.** A planning doc is not a substitute for `dotnet add package` + IntelliSense.

2. **Editor plugin behavior requires actual editor verification, not just build verification.** Build green ≠ plugin enabled. This is a checklist item for every editor panel story going forward.

3. **The stub-first architecture approach is a validated, repeatable pattern.** Lock public contracts and isolation boundaries before implementation stories. Carry forward to Epic 2 Story 2.1.

4. **One extendable validation lane is better than multiple purpose-built scripts.** `validate-foundation.py` scaled across all 10 stories. Do not create parallel validation scripts in Epic 2.

5. **Pre-epic backlog ordering is a gate, not a nice-to-have.** The 2026-04-11 sprint change revealed that architectural sequencing decisions in planning directly constrain implementation feasibility.
