## Successes and Strengths

### 1. Critical Preparation Items Executed — API Inspection Pattern Applied

Epic 1 identified that architecture documentation had assumed `DbConnection` was a concrete class when the SDK exposed `IDbConnection`. The lesson: inspect external API surfaces before writing acceptance criteria.

Story 2.2 did exactly this. Dev notes document the confirmed `DbConnectionBuilder` method signatures, the `OnConnect` callback delegate signature, and the `SpacetimeDB.Identity` struct behavior — all verified before ACs were finalized. The acceptance criteria then matched the actual SDK.

**Evidence:** Story 2.2 dev notes section "Confirmed SpacetimeDB SDK API (SpacetimeDB.ClientSDK 2.1.0)" — `WithToken()`, `OnConnect` delegate shape, identity parameter index — all confirmed before implementation began.

### 2. `ITokenStore` Concrete Implementations Delivered

Epic 1's highest-severity debt item (HIGH): `ITokenStore` defined in Story 1.3 but no concrete implementation shipped. Epic 2 Story 2.1 resolved this completely:

- `MemoryTokenStore` — field-backed, process-lifetime, no platform dependencies
- `ProjectSettingsTokenStore` — persists to Godot `ProjectSettings` under `spacetime/auth/token`
- `TokenRedactor` — prevents token values from appearing in logs

This was a prerequisite for all three subsequent auth stories. C2 from Epic 1 retro: ✅ completed.

### 3. Session Restore Through Existing Infrastructure (Story 2.3)

Story 2.3 delivered full session restoration in a single method modification. The design is notable: rather than creating a new code path, the restored token was injected through the identical `settings.Credentials → WithToken()` route that explicit credentials use. This meant `ConnectionAuthState.TokenRestored`, `StoreTokenAsync(token)`, and `WithToken()` injection all worked without changes — Story 2.3 was free.

**Evidence:** Story 2.3 file list: 3 files changed (1 connection service, 1 docs, 1 test file). No new production files created.

### 4. Test Count Growth — Contract Coverage Expanding

The test suite grew from 527 to 665 tests across the epic. Each story added static file/content contract tests covering:
- File existence and structure
- Property and method presence
- Behavioral invariants (isolation boundary checks, null-safety guards)
- Regression guards for all prior stories

The `pytest -q` full-suite gate caught no regressions across the epic.

### 5. Failure Taxonomy Completed (Story 2.4)

Story 2.4 completed the auth failure contract established in the architecture:

| Failure | How Surfaced | Auth State | Recovery |
|---------|-------------|------------|----------|
| Invalid settings | `ArgumentException` thrown | N/A | Fix settings |
| Stored token rejected | `ConnectionStatus` event | `TokenExpired` | `ClearTokenAsync()` + reconnect |
| Explicit credentials rejected | `ConnectionStatus` event | `AuthFailed` | Update `Credentials` + reconnect |
| Network failure (anonymous) | `ConnectionStatus` event | `None` | Retry `Connect()` |

Recoverable runtime failures and unrecoverable programming faults are now clearly separated at the public API surface.

## Challenges and Growth Areas

### 1. Every Story Required a Senior Review Fix

All four stories shipped clean builds and green test suites, but every story had findings during senior review. The two HIGH findings were functionally significant:

**Story 2.1 HIGH:** `ProjectSettingsTokenStore` wrote tokens via `SetSetting()` but never called `ProjectSettings.Save()`. Tokens would appear to be stored but would be lost on process exit — exactly the opposite of the documented behavior.

**Story 2.3 HIGH:** Token restoration injected a recovered token into `settings.Credentials` but didn't clear it after `_adapter.Open()`. A caller reusing the same `SpacetimeSettings` resource would find the restored token still present, making `Settings.TokenStore?.ClearTokenAsync()` ineffective for future connects.

**Root cause pattern:** Both HIGHs involve stateful side-effects that are correct in the happy path but incorrect under reuse/lifecycle conditions. The test suites were excellent at contract checking (file content, structure, API presence) but couldn't verify runtime persistence or lifecycle edge cases through static analysis alone.

**Impact:** Neither HIGH would have been caught by `pytest -q`. They would have required integration tests or actual runtime verification. The senior review step caught them — but this is a systemic gap in the testing strategy.

### 2. Five Epic 1 Action Items Rolled Over Unaddressed

The Epic 1 retrospective committed to five items that were not addressed in Epic 2:

| Item | Description |
|------|-------------|
| PA2/T1 | Document `ReconnectPolicy` defaults in `docs/connection.md` |
| PA3/T2 | Verify editor panels 1.7–1.9 in actual Godot 4.6.2 editor |
| P1 | Editor panel verification checklist for future panel stories |

The critical preparation items (C1, C2) were completed — showing the team can execute on non-optional dependencies. The optional/medium-priority items were not picked up.

**Root cause:** No specific owner was assigned to the non-blocking prep tasks during Epic 2. They were framed as "can happen during Story 2.1" (parallel tasks) but no one picked them up in parallel.

**Impact:** `docs/connection.md` still lacks `ReconnectPolicy` documentation. Editor panels from Stories 1.7–1.9 are still unverified in the actual Godot 4.6.2 editor. The `ConnectionAuthStatusPanel` was modified in Stories 2.2 and 2.4 but was also not editor-verified.

### 3. Three Deferred Story 2.2 [LOW] Review Items

Three LOW findings from Story 2.2 were not fixed and were carried forward:

1. **`ConnectionAuthState.AuthRequired` is dead code** — defined in the enum and has a panel arm, but no state machine code path ever emits it. The default `_` arm already handles it, making the `AuthRequired` arm unreachable.

2. **`OnConnectError` classifies all errors as `AuthFailed` when credentials provided** — network errors and server-offline scenarios produce the same `AuthFailed` state as credential rejection, even though AC 3 specifies "credential rejection" specifically.

3. **`AUTH FAILED` panel guidance for token-store path** — partially addressed in Story 2.4 (a `TokenExpired` arm was added), but the original LOW item about recovery text accuracy for the token-clearing path was only partially resolved.

These items are low severity individually but compound into a more confusing auth failure surface than the architecture intends.

### 4. Reflection-Based SDK Adapter Pattern Is Fragile Dependency

Story 2.2's identity capture required an expression-tree reflection approach:
```csharp
var identityExpression = parameterExpressions[1];
var toStringMethod = identityExpression.Type.GetMethod("ToString", Type.EmptyTypes) ...
```

This approach hard-codes the parameter index (`parameterExpressions[1]` for identity, `parameterExpressions[^1]` for token) based on the observed `SpacetimeDB.ClientSDK 2.1.0` callback shape. If the SDK changes the `OnConnect` delegate signature (reordering parameters, adding parameters before the existing ones), this adapter will break silently or throw at runtime.

The existing `test_story_1_4_adapter_boundary.py` isolation test confirms no `SpacetimeDB.*` references escape, but doesn't verify the reflection assumptions remain valid.

## Key Insights and Learnings

1. **Static contract tests can't catch runtime lifecycle bugs.** Both HIGH findings involved stateful lifecycle issues (`ProjectSettings.Save()` not called; restored credentials not cleared). The test suite verified structural invariants perfectly but couldn't exercise runtime state transitions. Integration or runtime-level tests are needed to close this gap.

2. **The "can happen in parallel" prep task framing doesn't work without an assigned owner.** PA2, PA3, P1 were all framed as parallel tasks that "can happen during Story 2.1" but had no named owner and no checkpoint. They rolled over unaddressed. Future parallel prep tasks need an owner and a verification gate.

3. **The stub-first pattern from Epic 1 scaled perfectly into Epic 2.** `ITokenStore` (1.3), `ConnectionStateMachine` (1.9), `SpacetimeSdkConnectionAdapter` (1.9) were all ready for extension. Story 2.2's `WithToken()` injection, Story 2.3's token restoration, and Story 2.4's `_restoredFromStore` field all built cleanly on top of the existing boundaries with no structural churn.

4. **A single-day epic delivery reveals implementation cadence maturity.** All four Epic 2 stories shipped on 2026-04-14. The boundary definitions (2.1) feeding credential injection (2.2) feeding restoration (2.3) feeding failure recovery (2.4) is a well-ordered dependency chain — each story consumed only what was already there.

5. **API inspection before ACs is now a confirmed repeatable practice.** Epic 1 identified the lesson; Epic 2 executed it. Story 2.2's explicit SDK API documentation in dev notes directly prevented a mid-implementation discovery of the type that caused Story 1.1's 3-cycle spec change.

6. **Dead code in the auth surface should be resolved before it propagates.** `ConnectionAuthState.AuthRequired` was defined in Story 2.2 for completeness but has no emission path. If it stays dead code through Epic 3, it will be present in released documentation and panel UI without ever being meaningful, confusing adopters.
