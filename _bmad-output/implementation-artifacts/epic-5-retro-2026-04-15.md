## Challenges and Growth Areas

### 1. Test Assertion Weakness (5/6 Stories)

**Pattern**: Initial test files used broad token-presence checks (`assert "demo/README.md" in content`) instead of ordered placement or contract assertions. Senior review expanded every test suite.

**Impact**: Tests passed for structurally wrong content (wrong lifecycle order, wrong API casing, missing required sections) because they only checked for string presence anywhere in the file.

**Root cause**: Test design was done after content design, and the "it looks complete" heuristic replaced rigorous contract specification.

### 2. Documentation Accuracy on First Pass (4/6 Stories)

**Pattern**: Lifecycle output ordering, API casing, and doc completeness repeatedly required correction before approval.

- Story 5.2: `Session identity` documented before `CONNECTED` in output sequence — contradicted actual runtime
- Story 5.2: `GetRows("smoke_test")` — wrong casing, silent runtime crash
- Story 5.3: Programming-fault comparison path missing from troubleshooting section
- Story 5.5: 11 MEDIUM findings — auth boundary, intro sourcing, subscription cleanup not adequately contracted

**Root cause**: Content written before contract — no pre-implementation specification of what exact strings, sequences, or API shapes must appear.

### 3. Story 5.5 — 11 Medium Findings

While zero HIGHs is good, 11 MEDIUMs in a single story represents a very thin initial test file for an eight-section document. The `P2` docs-precision process item from Epic 4 was not consistently applied to the most complex doc in the epic.

## Key Insights and Lessons Learned

### Insight 1: Documentation Epics Require the Same Contract Discipline as Code Epics

A string assertion `"See Also" in content` and a unit test `assert result == expected` fail the same way when the implementation is structurally wrong but superficially correct. The test discipline that eliminated HIGH findings from code stories must be applied identically to documentation stories.

**Action**: Pre-implementation test design step — define contract assertions before writing doc content.

### Insight 2: The `GetRows` PascalCase Edge Is a Recurring Pattern

Four epics have seen snake_case/PascalCase confusion at the SDK boundary. The 5.2 `GetRows("smoke_test")` bug is the most dangerous instance — it produces a silent runtime exception. No static test can catch this without a dynamic lifecycle test.

**Action**: D3 (dynamic lifecycle tests) is the permanent fix. Until then, `docs/runtime-boundaries.md` must explicitly call out the PascalCase requirement in a prominent location.

### Insight 3: `ProjectSettingsTokenStore` Externalization Is Undocumented

`ProjectSettingsTokenStore` is `internal sealed`. External developers cannot instantiate it. They must implement `GodotSpacetime.Auth.ITokenStore` themselves. This is documented only in Story 5.2 dev notes — not in any published doc. Before Epic 6 ships a public release, this must be explicit in `docs/runtime-boundaries.md`.

### Insight 4: Retro Action Items Without Story Anchors Do Not Get Done

P1 succeeded because it was procedural (embedded in a checklist). D1–D4 failed because they were retro commitments without implementation homes. Every carry-forward debt item must become an explicit subtask in an Epic 6 story.
