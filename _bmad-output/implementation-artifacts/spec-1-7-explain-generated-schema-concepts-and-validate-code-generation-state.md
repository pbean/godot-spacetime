# Story 1.7: Explain Generated Schema Concepts and Validate Code Generation State

Status: done

## Story

As a Godot developer,
I want clear explanations of generated schema concepts plus visible code-generation status and recovery guidance,
so that I can understand the generated surface and diagnose code-generation problems from the supported editor workflow.

## Acceptance Criteria

1. **Given** generated bindings and configured code-generation inputs, **when** I inspect the supported code-generation docs, **then** the workflow explains which generated types correspond to tables, reducers, events, and other exposed schema concepts.
2. **And** the code-generation configuration and validation surface shows module source, output location, last generation result, and the next recovery action when configuration or generation fails.
3. **And** the status messaging uses explicit text labels, visible focus, and keyboard-accessible navigation rather than color-only cues.
4. **And** the committed code-generation surface remains usable in narrow, standard, and wide editor panel widths.

## Deliverables

This story produces five concrete outputs:

1. **`docs/codegen.md` extension** — New "Generated Schema Concepts" section explaining table row types, table handles, reducer proxies, and the module binding registry, referencing the actual `smoke_test` generated files.
2. **`addons/godot_spacetime/src/Editor/Codegen/CodegenValidationPanel.cs`** — C# editor panel that shows module source path, output location, generation status (text label, not color-only), and recovery guidance command when bindings are missing.
3. **`addons/godot_spacetime/src/Editor/Codegen/CodegenValidationPanel.tscn`** — Godot scene for the validation panel, used as a bottom-panel dock registered by the plugin.
4. **`addons/godot_spacetime/GodotSpacetimePlugin.cs` update** — Register `CodegenValidationPanel` as a bottom panel with `AddControlToBottomPanel` during `_EnterTree`, remove it during `_ExitTree`.
5. **`scripts/compatibility/support-baseline.json` update** — Add new required paths for `Editor/`, `Editor/Codegen/`, `.cs`, and `.tscn` files, plus a `line_checks` entry confirming the schema concepts section exists in `docs/codegen.md`.
6. **`tests/test_story_1_7_schema_concepts.py`** — pytest suite verifying the schema-concepts docs section, panel file existence, explicit text labels in panel code, updated support-baseline entries, and regression guards from Story 1.6.

No changes to `addons/godot_spacetime/src/Public/` or `addons/godot_spacetime/src/Internal/`. No changes to `demo/generated/smoke_test/` (read-only artifacts stay untouched).

### What the Generated Files Are (for the docs section)

The actual files produced by `spacetime generate --lang csharp` from the `smoke_test` module are:

```
demo/generated/smoke_test/
├── SpacetimeDBClient.g.cs   — module binding registry (RemoteTables, RemoteReducers, etc.)
├── Reducers/
│   └── Ping.g.cs            — reducer proxy: Ping() call method + OnPing event + Reducer.Ping args
├── Tables/
│   └── SmokeTest.g.cs       — table handle: RemoteTables.SmokeTestHandle + IdUniqueIndex
└── Types/
    └── SmokeTest.g.cs       — table row type: SmokeTest class with DataMember attributes
```

All generated types live in the `SpacetimeDB.Types` namespace. The generator uses `SpacetimeDB.ClientSDK 2.1.0` patterns:

- **Table row type** (`Types/SmokeTest.g.cs`): `[SpacetimeDB.Type] [DataContract] public sealed partial class SmokeTest` with `[DataMember(Name = "id")] public uint Id` and `[DataMember(Name = "value")] public string Value`. This is the data class — one instance per cached row.
- **Table handle** (`Tables/SmokeTest.g.cs`): `RemoteTables.SmokeTestHandle` extends `RemoteTableHandle<EventContext, SmokeTest>`. Contains `IdUniqueIndex` for `Id`-keyed lookups. Accessed via `conn.Db.SmokeTest` (where `conn.Db` is the `RemoteTables` instance). The handle is the cache access layer.
- **Reducer proxy** (`Reducers/Ping.g.cs`): `RemoteReducers.Ping()` method calls `conn.InternalCallReducer(new Reducer.Ping())`. `RemoteReducers.OnPing` is a `PingHandler` event that fires when the server acknowledges the call. `Reducer.Ping` implements `IReducerArgs` with `ReducerName = "ping"`.
- **Module binding registry** (`SpacetimeDBClient.g.cs`): `RemoteTables`, `RemoteReducers`, `RemoteProcedures`, and `IRemoteDbContext` interface. `EventContext` carries the event that triggered a callback. This file is the generated entry point — it ties together the entire module surface.

### Editor Panel — Key Implementation Decisions

**No CLI invocation from the panel.** UX2 explicitly requires the panel to support the code-first SDK model rather than replace runtime API flows. The panel is a status indicator only — it reads the file system, not the network or the CLI.

**Output path resolution.** The panel checks `ProjectSettings.GlobalizePath("res://") + "demo/generated/smoke_test/"` for `.cs` files. This is a hardcoded default for v1. Story 1.9 may later add a configurable path via `SpacetimeSettings`.

**Explicit text labels over color.** NFR24 requires text-based status, not color-only. The three status strings (`StatusOk`, `StatusMissing`, `StatusNotConfigured`) cover all states. Do not add `_statusLabel.AddThemeColorOverride("font_color", ...)` as the sole distinguisher — pair any color changes with the text label.

**Narrow panel width.** NFR23 requires the panel to work in narrow Godot editor panel widths. Use `VBoxContainer` as the root with `CustomMinimumSize = new Vector2(200, 0)`. Label text may truncate in narrow view; this is acceptable as long as the status text is not lost. Do NOT use `HBoxContainer` for the status row — vertical stacking degrades better in narrow widths.

**`#if TOOLS` guard.** All editor code must be inside `#if TOOLS` blocks. Failing to wrap `CodegenValidationPanel.cs` will cause compile errors in non-tool builds.

**Plugin registration pattern.** `GodotSpacetimePlugin.cs` uses `AddControlToBottomPanel` in `_EnterTree` and `RemoveControlFromBottomPanel` followed by `QueueFree()` in `_ExitTree`. This is Godot's documented pattern for adding editor bottom panels and must be followed exactly to avoid plugin lifecycle crashes.

### `CodegenValidationPanel.tscn` — Minimal Valid Format

The `.tscn` file must be a valid Godot scene file. Minimal example:

```
[gd_scene load_steps=2 format=3]

[ext_resource type="Script" path="res://addons/godot_spacetime/src/Editor/Codegen/CodegenValidationPanel.cs" id="1_aaaaa"]

[node name="CodegenValidationPanel" type="VBoxContainer"]
script = ExtResource("1_aaaaa")
```

The `uid` field is optional in `format=3`; Godot will assign one on first open. The `load_steps=2` counts the scene node + the ext_resource. If the build fails due to a missing UID, open the scene in the Godot editor once to let it assign a UID and re-commit.
